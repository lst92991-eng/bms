# BMS 项目并行工作机制

记录日期：2026-06-22

本文档规定本项目如何使用主线程和多个子 agent 并行工作。目标是防止上下文过长后跑偏、防止主线程因长期承载细节而过早压缩、防止压缩恢复后丢失方向、防止旧项目假设污染新项目、防止硬件事实和业务策略混写、防止一次生成过多代码导致无法审核。

## 1. 总原则

- 主线程是唯一调度者、唯一事实整合者、唯一最终决策者，但不是唯一代码编写者。
- 子 agent 只能处理被分配的窄任务，不能自行扩大范围。
- 子 agent 的输出默认是“待核验材料”，不是最终事实。
- 最基础的代码 worker 可以在任务卡授权的写范围内直接写代码。
- 主线程不长期保存大段 datasheet、旧代码或 agent 长报告；这些内容必须沉淀到 `docs/agent_reports`、`docs/pre`、`docs/review` 或 `docs/state`。
- 主线程每完成一个门禁，必须更新 `docs/state/main_thread_checkpoint.md`，让上下文压缩后可以恢复方向。
- 每个代码模块必须先有 `pre` 文档，再写一对 `.c/.h`，最后写 `review` 文档。
- 每一轮只允许推进一个模块，不允许同一轮同时生成多个 INT/COM/APP 模块。
- 旧项目只提供风格参考，不能作为新硬件事实来源。
- 官方手册、当前网表、原理图、用户确认的实物改焊信息必须优先于旧项目代码。

## 2. 角色划分

### 2.1 主线程

职责：

- 读取 README、`docs/rules`、当前模块 `pre` 和用户最新要求。
- 拆分子任务，并限制每个子 agent 的输入、输出和禁止事项。
- 合并子 agent 结论，标明直接事实、推断和待确认项。
- 决定是否进入下一道门禁。
- 最终写入或合并 `docs/pre`、`docs/review`、`docs/logic`、`docs/state` 和代码补丁。
- 维护当前模块的任务卡、写权限、冻结状态和压缩恢复状态。
- 向用户汇报当前阶段、阻塞项和需要审核的最小内容。

禁止：

- 让子 agent 直接决定 API。
- 让子 agent 直接跨层实现业务。
- 让没有任务卡的 worker 写代码。
- 因为旧项目存在某个函数就默认新项目也需要同名函数。
- 把长篇 agent 输出原样塞进主线程上下文反复推理。
- 压缩恢复后不读取状态文件就继续执行。

### 2.2 硬件事实 agent

输入：

- 当前新项目网表、原理图、`docs/rules/hardware_rules.md`、用户硬件说明。

输出：

- 器件型号、引脚、网络名、上拉/下拉、默认电平、硬件风险、待确认项。

禁止：

- 设计业务策略。
- 设计 public API。
- 写代码。
- 用旧项目硬件覆盖新项目硬件。

### 2.3 Datasheet/Register agent

输入：

- 官方 datasheet、TRM、软件开发指南、EVM guide。

输出：

- 寄存器地址、位域、单位换算、默认值、安全禁止项、引用来源。

禁止：

- 根据网文或记忆补寄存器。
- 忽略未确认的位域。
- 写业务策略。
- 写代码。

### 2.4 旧项目风格 agent

输入：

- 旧项目 `Core`、`MDK-ARM/Int`、`MDK-ARM/Com`、`MDK-ARM/App`。

输出：

- 分层结构、命名风格、注释风格、可继承点、不可继承风险。

禁止：

- 把旧 BQ76930 寄存器当作新 BQ76952 寄存器。
- 把旧 STM32F1 外设接口当作新 STM32G0 接口。
- 直接提出新项目业务策略。

### 2.5 API 设计 agent

输入：

- 主线程合并后的硬件事实、寄存器事实、旧风格总结和模块边界。

输出：

- public API 草案、结构体草案、状态码草案、每个接口存在的原因、删掉哪些接口及原因。

禁止：

- 写 `.c/.h`。
- 超过模块预算。
- 设计 App 层业务。

### 2.6 代码 worker agent

输入：

- 已冻结的 `pre` 文档。
- 主线程签发的任务卡。
- 明确的写入文件、函数范围、禁止修改范围和验收标准。

输出：

- 指定模块的一对 `.c/.h` 草案，或指定范围内的补丁。
- 如果任务卡只授权一个方法，则只能实现该方法及必要的 `static` 辅助函数。
- 写入后的自查说明。

权力：

- 叶子代码 worker 有写代码权力。
- 写代码权力只在任务卡授权的文件和方法范围内有效。
- 多个 worker 可以并行写代码，但写集合必须互斥；如果无法互斥，必须串行。

禁止：

- 修改未授权文件。
- 修改未授权函数。
- 同时实现多个模块。
- 回滚其他人改动。
- 绕过 `pre` 文档新增 public API。
- 修改 `docs/rules`、`docs/state`，除非任务卡明确授权。
- 为了让自己的方法编译而擅自改变其他模块 API。

### 2.7 Review/Safety agent

输入：

- 当前 diff、`pre` 文档、`hardware_rules.md`、本机制文档。

输出：

- blocking 问题、硬件约束破坏、API 膨胀、跨层问题、错误处理缺口、验证缺口。

禁止：

- 直接重写代码。
- 只做风格点评而忽略硬件安全。

## 3. 门禁流程

每个模块必须按以下顺序推进。

### 3.1 Source Gate

通过条件：

- 明确当前模块名称。
- 明确本轮只做哪个模块。
- 明确读取了哪些文件。
- 明确旧项目只能作为风格参考。

输出：

- 当前模块资料清单。
- 不参与本轮的资料清单。

### 3.2 Hardware Fact Gate

通过条件：

- 器件型号、引脚、网络名、极性、默认态已记录。
- 对电源、充电、保护类模块，安全态已记录。
- 所有硬件冲突和不确定项已写明。

输出：

- 硬件事实表。
- blocking 风险。
- bring-up 前检查项。

### 3.3 Register/Datasheet Gate

通过条件：

- 所需寄存器、位域、单位换算来自官方资料。
- 未确认位域不得进入代码。
- 安全禁止配置已列出。

输出：

- 寄存器表。
- 位域说明。
- 公式和取值范围。
- 禁止写入项。

### 3.4 API Gate

通过条件：

- public API 默认不超过 12 个。
- 每个 public API 有明确用途、输入、输出、错误返回和“不做什么”。
- 没有重复语义接口。
- 没有跨层业务逻辑。
- 用户审核通过。

输出：

- `docs/pre/<module>_pre.md`。
- 用户确认记录或待修改问题。

### 3.5 Implementation Gate

通过条件：

- 只写当前模块的一对 `.c/.h`，或任务卡授权的单个方法/局部代码块。
- 每个 worker 的写集合互斥，已记录到 `docs/state/main_thread_checkpoint.md`。
- 所有 I/O 函数返回状态。
- ISR 只置标志，不访问 I2C/SPI/CAN 等慢外设。
- 不默认 `printf`。
- 不新增 App 业务策略。
- 对 SC8815、BQ76952 这类安全相关芯片，所有写寄存器路径经过 guard。

输出：

- 当前模块 `.c/.h`。
- 实现自查记录。

### 3.6 Review Gate

通过条件：

- Review 先列 blocking 问题。
- 已检查硬件约束、API 边界、错误返回、跨层访问、代码膨胀。
- 已记录无法验证项。

输出：

- `docs/review/<module>_review.md`。
- 修复清单。

### 3.7 Freeze Gate

通过条件：

- `pre`、代码、`review` 三者一致。
- 用户没有未处理的 blocking 审核意见。
- 最小编译或静态检查已执行；如果不能执行，必须说明原因。

输出：

- 模块首版冻结记录。
- 下一个模块建议。

## 4. 子任务拆分粒度

允许拆分：

- 单个芯片的硬件事实读取。
- 单个芯片的寄存器表摘录。
- 单个旧模块的风格分析。
- 单个 public API 的必要性审查。
- 单个 public API 的实现。
- 单个 `static` 私有方法的实现。
- 单个寄存器 guard 方法的实现。
- 单个换算公式或查表函数的实现。
- 单个 `.c/.h` 的 review。

禁止拆分：

- 一个 agent 同时负责 SC8815、BQ76952、OLED、EEPROM。
- 一个 agent 同时做硬件事实、API 设计和代码实现。
- 一个 agent 同时改 INT、COM、APP。
- 一个 agent 根据不完整资料先写代码再补文档。

## 5. 固定输出格式

每个子 agent 最终必须按以下格式输出：

```text
读取文件：
- ...

直接事实：
- ...

推断：
- ...

不确定项：
- ...

对当前模块的影响：
- ...

建议进入下一步：
- 是/否，原因...

文件改动：
- 无
```

如果是代码 worker，必须额外输出：

```text
写入文件：
- ...

未实现内容：
- ...

偏离 pre 文档之处：
- 无/...

任务卡编号：
- ...

写权限范围：
- ...
```

## 6. 事实优先级

当资料冲突时，按以下顺序判定：

1. 用户确认的当前实物、改焊、硬件问题。
2. 当前新项目网表和原理图。
3. 官方 datasheet、TRM、软件开发指南、EVM guide。
4. 新 CubeMX `.ioc` 和 HAL 生成代码。
5. 旧项目代码。
6. 旧对话记忆或未落文档内容。

安全相关冲突默认阻塞，不允许静默选择一个版本继续写代码。

## 7. 防跑偏检查表

每次主线程准备签发任务卡、写文档、合并代码或恢复上下文后，必须检查：

- 本轮是否只推进一个模块。
- 当前结论是否能追溯到文件或用户确认。
- 是否把旧项目寄存器、MCU 外设、GPIO 名称误搬到新项目。
- 是否把 App 业务策略写进 INT。
- 是否新增了未经 `pre` 审核的 public API。
- 是否遗漏安全默认态和错误返回。
- 是否存在“为了方便”而引入的大框架或通用抽象。
- 是否有无法验证但没有记录的假设。
- 当前 worker 是否有任务卡。
- 当前 worker 写集合是否和其他 worker 冲突。
- 主线程状态文件是否记录了当前门禁、当前模块、下一个动作和未解决问题。

## 8. 主线程上下文保护机制

主线程必须保持“瘦上下文”。大资料由子 agent 消化，主线程只保存可验证结论和决策。

### 8.1 持久状态文件

主线程状态固定写入：

- `docs/state/main_thread_checkpoint.md`

该文件必须包含：

- 用户最新目标。
- 当前模块。
- 当前门禁。
- 已批准事实。
- 已拒绝假设。
- 当前活动任务卡。
- 当前写权限锁。
- 未解决问题。
- 下一步唯一动作。
- 最近一次用户审核结论。

主线程每完成以下动作之一，必须更新状态文件：

- 用户改变目标。
- 进入或退出一个门禁。
- 签发任务卡。
- 收到 worker 代码结果。
- 合并代码。
- 写入 review。
- 发现 blocking 风险。
- 准备结束当前回合。

### 8.2 压缩恢复门禁

主线程发生上下文压缩、恢复、长时间中断或用户指出“跑偏”后，必须先执行 Rebase Gate。

Rebase Gate 通过条件：

- 重新读取 `README.md`。
- 重新读取 `docs/rules/parallel_work_mechanism.md`。
- 重新读取 `docs/rules/hardware_rules.md`。
- 重新读取 `docs/state/main_thread_checkpoint.md`。
- 重新读取当前模块 `docs/pre/<module>_pre.md`，如果存在。
- 对比用户最新消息和状态文件，若冲突，以用户最新消息为准并更新状态文件。
- 明确“现在只做哪一步”，然后才能继续。

Rebase Gate 禁止：

- 依赖压缩前记忆继续写代码。
- 直接继续执行旧计划。
- 把已关闭或超时 agent 的结论当作已确认事实。

### 8.3 主线程上下文预算

主线程不能长期保留以下内容：

- datasheet 大段摘录。
- PDF 大段文本。
- 旧项目源码大段内容。
- worker 的完整实现解释。
- 多个模块的 API 草案。

这些内容必须写入文件，再由主线程读取摘要：

- agent 长报告写入 `docs/agent_reports/<module>_<role>_<date>.md`。
- 寄存器事实写入 `docs/pre/<module>_registers.md` 或当前模块 `pre`。
- 当前决策写入 `docs/state/main_thread_checkpoint.md`。

### 8.4 主线程替代机制

主线程不是固定不变的长期记忆容器，而是可以换代的调度角色。替代机制的目标是：当 Codex 明确提示上下文过长、即将压缩、已经压缩或由新模型/新线程接续时，让新主线程只依赖落地文件恢复方向，而不是凭残缺记忆继续。

替代机制的硬触发以 Codex 的上下文提示为准。没有 Codex 提示时，主线程不因为主观判断“上下文可能太长”就主动换代；此时只能刷新交接包和状态文件。

硬触发条件：

- Codex 明确提示上下文过长、接近上限、即将压缩或需要交接。
- 当前对话出现上下文压缩、恢复、handoff summary、`Another language model...` 之类的接续提示。
- Codex 或系统提示当前主线程已由新的模型/线程接班。
- 用户明确要求“替换主线程”“重新接班”，且主线程先按本节更新交接包。

预备信号（只要求刷新交接包，不自动换代）：

- 主线程已经连续推进两个以上模块或一个模块内多个门禁。
- 主线程为了当前任务读取了大段 datasheet、旧代码、PDF 抽取文本或多个 agent 长报告。
- 当前活动 worker 超过 2 个，或写权限锁超过 2 把。
- 准备从一个层级切换到另一个层级，例如 `INT -> COM`、`COM -> APP`。
- 用户指出“跑偏”“上下文太长”“重新接上”“继续”但状态文件落后。
- 主线程无法在短上下文里清楚说明当前模块、门禁、写锁、唯一下一步。

出现预备信号时，主线程必须先补齐 `docs/state/main_thread_checkpoint.md` 的交接包，并减少上下文承载；只有同时出现硬触发条件时才执行正式 Replacement Freeze/Gate。

出现 Codex 硬触发后，替代前当前主线程必须执行 Replacement Freeze：

- 停止签发新的任务卡。
- 不再做代码编辑，除非正在写的是替代交接文档本身。
- 更新 `docs/state/main_thread_checkpoint.md`，写入主线程世代、替代原因、当前模块、当前门禁、唯一下一步。
- 写清所有活动 agent、任务卡、写权限锁、锁状态和锁释放条件。
- 写清当前没有合并的 worker 结果；未合并结果只能作为“待核验材料”。
- 写清最近一次用户要求，若与状态文件旧目标冲突，以用户最新要求为准。
- 如果有活动 worker 正在写代码，默认不允许新主线程继续发新锁，必须先等待、撤销或合并这些锁。

出现压缩恢复、handoff summary 或新模型接续提示后，新主线程接班必须执行 Replacement Gate：

1. 重新读取 `README.md`、`docs/rules/parallel_work_mechanism.md`、`docs/rules/hardware_rules.md`、`docs/state/main_thread_checkpoint.md`。
2. 重新读取当前模块 `pre`、当前模块 `review`，如果已经存在。
3. 检查 `main_thread_checkpoint.md` 中的主线程世代和唯一下一步。
4. 检查工作区是否有未合并改动；不明来源改动不得回滚，必须记录并避让。
5. 检查所有活动任务卡和写权限锁；未确认释放的锁不得被重复分配。
6. 把已超时、失败、关闭或身份不明的 agent 输出降级为“参考材料”，不得直接作为事实。
7. 在 `main_thread_checkpoint.md` 写入“接班确认”，包括新主线程世代、接班时间、接班后唯一动作。
8. 只有接班确认完成后，才能继续签发任务卡或写代码。

替代机制禁止：

- 新主线程跳过状态文件，直接沿用旧上下文计划。
- 在还有未决写锁时，给另一个 worker 分配同一文件或同一函数。
- 把旧主线程的口头记忆当成已批准事实。
- 把 agent 的长输出重新塞回主线程上下文反复推理。
- 在替代交接未完成时跨模块推进。
- 没有 Codex 上下文提示时，仅凭主线程主观判断就正式换代。

### 8.5 主线程交接包

每次 Codex 提示上下文可能压缩、已经压缩或需要新线程接续前，`docs/state/main_thread_checkpoint.md` 必须足以构成交接包。没有 Codex 提示但出现预备信号时，也必须刷新该交接包，但不自动换代。交接包至少包含：

- `主线程世代`：例如 `G1`、`G2`。
- `替代原因`：主动瘦身、用户纠偏、上下文压缩、长时间中断等。
- `最近用户目标`：一句话说明当前用户最新要求。
- `当前模块/门禁`：只允许一个当前模块。
- `冻结事实`：已经通过门禁的事实列表。
- `拒绝假设`：不能再走的错误方向。
- `活动任务卡`：任务卡编号、agent、状态、允许写范围。
- `写权限锁`：文件级或函数级锁，包含锁持有者和释放条件。
- `未合并输出`：哪些 worker 输出还未审核。
- `唯一下一步`：接班后只能做这一件事。
- `禁止下一步`：明确列出不能顺手做的事。

如果交接包无法让新主线程在 5 分钟内恢复方向，说明主线程已经承载了过多隐性上下文，必须先补交接包，不能继续写代码。

## 9. 任务卡与写权限

主线程给代码 worker 签发任务卡后，worker 才能写代码。

任务卡模板：

```text
任务卡编号：
模块：
当前门禁：
输入文件：
允许读取：
允许写入：
允许修改的函数/代码块：
禁止修改：
必须遵守的 pre 条款：
```

写权限规则：

- 一个任务卡只能对应一个模块。
- 一个叶子 worker 优先只实现一个 public API 或一个 `static` 方法。
- 若实现一个方法需要新增私有辅助函数，必须说明原因，且不得新增 public API。
- 多个 worker 并行时，`允许写入` 和 `允许修改的函数/代码块` 必须互斥。
- worker 修改完成后，主线程只做合并和 review，不重写 worker 的代码，除非 review 发现 blocking 问题。
- 主线程合并前必须检查 worker 是否越权修改。

写权限锁登记必须写入 `docs/state/main_thread_checkpoint.md`。当任务数量增多时，可以再拆出 `docs/state/write_lock_registry.md`，但登记字段不能减少。

写权限锁字段：

- `锁编号`：通常等同任务卡编号。
- `持有者`：agent id 或主线程。
- `模块`：例如 `Int_SC8815`。
- `门禁`：Implementation、Review 等。
- `锁粒度`：文件级、函数级、代码块级。
- `允许写入文件`：完整路径列表。
- `允许修改函数/代码块`：函数名、宏区段或文档章节。
- `禁止写入文件`：至少列出同模块外文件、`docs/rules/*`、未授权 `docs/state/*`。
- `基线`：签发任务卡时的文件状态说明或 commit/diff 状态。
- `租约`：签发时间、预计返回时间、是否允许延期。
- `状态`：requested、granted、in_progress、returned、merged、released、revoked、paused。
- `释放条件`：代码合并、review 驳回、用户取消、超时撤销。

### 9.1 文件归属矩阵

为避免 agent 因角色不同而互相覆盖，默认文件归属如下：

| 文件/目录 | 默认写入者 | 是否可给 worker | 规则 |
| --- | --- | --- | --- |
| `docs/rules/*` | 主线程 | 否 | 只改项目机制和全局规则；用户纠偏时优先更新。 |
| `docs/state/*` | 主线程 | 极少 | 状态、写锁、交接包只由主线程维护；worker 只读。 |
| `docs/pre/*` | 主线程/API agent 草案后主线程落地 | 谨慎 | pre 冻结 API 和边界；代码 worker 不得改。 |
| `docs/agent_reports/*` | 子 agent | 是 | 只放事实报告或分析，不作为最终事实直接使用。 |
| `docs/review/*` | Review/Safety 或主线程 | 是 | review 可由只读 agent 生成；不得同时改代码。 |
| `Int/*_BSP.h` | Datasheet/Register worker 或主线程 | 串行 | 安全芯片 BSP 一次只允许一个写者。 |
| `Int/*.h` | 当前模块代码 worker | 串行优先 | public API 已冻结后才写；同一头文件不并行。 |
| `Int/*.c` | 当前模块代码 worker | 可函数级锁 | 只有函数边界稳定且无共享 static 状态时才拆分。 |
| `Com/*`、`App/*` | 对应层 worker | 当前阶段否 | 未进入对应层前不得写。 |
| 旧项目目录 | 无 | 否 | 只读风格参考，禁止修改。 |

如果某个文件不在矩阵中，默认归主线程，只读等待重新登记。

### 9.2 锁租约与心跳

写权限锁不是永久授权，而是租约：

- 主线程签发任务卡时必须给出预计返回点；可以是时间、门禁或“当前回合结束前”。
- worker 返回结果后，锁状态先变为 `returned`，不是 `released`。
- 主线程完成越权检查和 review 后，才能把锁标为 `merged` 或 `released`。
- worker 需要延期时，必须说明已修改文件、未完成函数、是否存在未保存风险。
- worker 超时、失败、429、关闭或无输出时，锁进入 `paused`，主线程必须明确写 `revoked` 或继续等待。
- `paused` 锁持有的文件不得分配给另一个 worker，除非主线程确认该 worker 没有可合并输出，并把锁标记为 `revoked`。

### 9.3 合并队列

主线程一次只处理一个 returned 锁，形成合并队列：

1. `returned`：worker 已返回，主线程尚未审查。
2. `inspecting`：主线程检查越权、API、硬件安全和静态问题。
3. `accepted`：通过检查，可以合并或保留当前工作区改动。
4. `needs_fix`：有 blocking 问题，回到同一 worker 或由主线程在同一写锁内修复。
5. `rejected`：越权或方向错误；不得直接合并。
6. `released`：锁结束，同一文件可以重新分配。

合并队列规则：

- 同一文件存在 `returned` 结果未审查时，不得再签发该文件的新锁。
- Review agent 默认只读；若需要修复，必须转为新的代码 worker 任务卡。
- 主线程不能同时审查两个 worker 对同一文件的改动。
- 如果合并发现 pre 不够清楚，先修 pre，再重新签发任务卡；不要在代码里扩大边界。

冲突规避规则：

- 默认采用文件级独占锁；同一 `.c` 或 `.h` 文件同时只给一个 worker 写。
- 只有文件结构稳定、函数边界清楚、且两个任务没有共享 static 状态时，才允许函数级锁。
- `docs/rules/*` 只允许主线程修改；worker 不得改规则文档。
- `docs/state/*` 默认只允许主线程修改；如果 worker 必须写状态，任务卡必须单独授权具体章节。
- 一个 worker 不得为了自己的代码能编译而修改另一个 worker 持锁的文件。
- 如果用户改变目标，所有未返回的写锁进入 `paused` 状态；新主线程必须重新确认是否继续有效。
- agent 失败、超时、429、关闭或输出不完整时，该锁不得自动视为释放；主线程必须在状态文件中显式写 `revoked` 或 `released`。
- 被撤销锁的迟到输出只能作为参考材料，不能直接合并。

合并前检查：

1. 检查 worker diff 是否只触及授权文件。
2. 检查 public API 是否仍等于 `pre` 文档批准范围。
3. 检查是否改动 `docs/rules`、`docs/state`、旧项目目录或其他模块。
4. 检查是否引入跨层业务逻辑、printf、FreeRTOS 或未授权外设。
5. 检查是否破坏安全 guard 或硬件默认安全态。
6. 如果发现越权修改，先记录 review blocking；不得顺手扩大任务范围来“修好它”。

主线程替代时的写锁规则：

- 新主线程接班后，第一件事是读取写锁登记。
- 未释放锁不得被重复签发。
- 不清楚来源的本地文件改动视为外部改动，不能回滚，只能避让或记录阻塞。
- 如果需要继续同一文件，必须先把旧锁标记为 `released`、`merged` 或 `revoked`。
- 替代期间不得并行签发新 worker；先让项目重新回到单一明确下一步。

## 10. 当前推荐启动顺序

在没有新 CubeMX 工程和 `.ioc` 的情况下，先做文档和低风险 INT：

1. 固化本并行工作机制。
2. 建立 `docs/state/main_thread_checkpoint.md`。
3. 固化 `docs/rules/coding_rules.md`。
4. 写 `docs/pre/int_gpio_board_pre.md`，只定义 LED、蜂鸣器、按键、BMS wake/shut 的 INT 边界。
5. 用户审核 GPIO API 后，给叶子代码 worker 签发任务卡。
6. 叶子 worker 按任务卡直接写 `Int_GpioBoard.c/.h` 中授权的方法。
7. Review/Safety agent 检查 worker 是否越权和是否破坏硬件边界。
8. 再进入 `Int_OLED`、`Int_EEPROM`。
9. `Int_SC8815` 必须先由 Datasheet/Register agent 产出寄存器表和 guard 清单。
10. `Int_BQ76952` 单独立项，不能复用旧 BQ76930 BSP。

如果用户要求优先验证充电链路，可以把 `Int_SC8815` 提前，但仍必须先完成 SC8815 的 `pre` 和寄存器门禁。
